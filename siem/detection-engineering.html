SIEM Track Authoritative Methodology

We migrate detections based on outcomes and coverage, not rule equivalence. If XDR reliably detects the same attack technique, the SIEM rule is retired. This page documents our authoritative methodology for RSA NetWitness â†’ Microsoft Sentinel migration.

Core Philosophy: Detection-Driven, Not Log-First

Our SIEM migration was detection-driven rather than log-first or 1-to-1 rule translation.

Instead of mechanically porting every RSA NetWitness rule, we evaluated detection intent and leveraged Microsoft's XDR platform for domains where built-in detections already provide strong coverage â€” endpoint, identity, email, and cloud app activity.

This allowed us to retire redundant detections, reduce ingestion cost, and avoid duplicating logic that XDR already handles natively. Sentinel was then used where it adds the most value: cross-domain correlation, non-Microsoft telemetry, and detections that require broader context beyond XDR.

ğŸ¯ Interview-Ready Answer: Detection Engineering Strategy

"After discovery from a legacy SIEM, my detection engineering strategy is outcome-driven rather than rule-driven."

  1. Normalize discovery into security outcomes, not existing rules
  2. Decide ownership for each outcome (XDR vs SIEM)
  3. Define log ingestion based on ownership decisions
  4. Design fewer, higher-fidelity detections focused on correlation
  5. Validate coverage using ATT&CK, SOC feedback, and real incidents

5Apply Clear Drop Criteria (This Is Key)

You DROP / DO NOT TRANSLATE an RSA rule if any of the following are true:

ğŸš« DROP CRITERION 4: Redundant Correlation
