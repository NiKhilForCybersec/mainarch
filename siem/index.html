SIEM Migration: RSA NetWitness ‚Üí Microsoft Sentinel

Track 1 Status: Complete ‚Äî Handover in Progress

The SIEM track started at M1 alongside EDR. Sentinel is now operational, log sources are onboarded, RSA is decommissioned, and the SOC is running on the new platform. Currently in final handover phase with knowledge transfer to internal security team.

Role Scope

Platform Engineering: Log source discovery, stakeholder coordination, ingestion strategy, Sentinel workspace setup, connector configuration, custom log onboarding, and operational readiness.

Detection Engineering: Implemented and enabled built-in detection rules identified in the detection translation catalog ‚Äî specifically those with direct native equivalents in Sentinel and M365 Defender. Translated and aligned detection use cases around endpoint telemetry, ensuring proper correlation between MDE, identity, and network signals within the SIEM.

Detection Engineering Team: Detailed custom KQL detection development, complex rule translation, ongoing threshold tuning, and fine-tuning handled by SOC analysts and dedicated detection engineers.

Current State

Sentinel + XDR: XDR-first architecture operational from Day 1. M365 Defender handles endpoint, identity, and email detection; Sentinel owns network, Azure, and higher-order correlation (~270 rules). RSA NetWitness decommissioned.

Log Sources: All critical sources onboarded via built-in connectors and custom DCRs.

SOC: Analysts trained and operational on M365 Defender unified queue (security.microsoft.com).

Detection Split: XDR handles ~350 detection types automatically; Sentinel handles ~270 rules for non-Microsoft sources and higher-order correlation.

Current Phase: Knowledge transfer and handover to internal security team.

XDR-First Approach

Key Insight: In the Microsoft ecosystem, correlation is layered ‚Äî XDR handles endpoint + identity + email correlation upstream, and Sentinel handles network + Azure + higher-order correlation. This means you don't rebuild detections XDR handles natively.

Result: ~270 Sentinel rules (not 500+), 12-month timeline, XDR correlation benefits from Day 1.

Project Timeline Context

Identity & Infrastructure Started First: Cloud foundation (AD ‚Üí Entra ID, Azure landing zone) was already underway before security tracks began. You can't migrate security tools until the cloud foundation is in place.

SIEM & EDR Started at M1: Once foundation was ready, RSA ‚Üí Sentinel and ESET ‚Üí MDE migrations began. MDE rollout started early to establish telemetry before detection implementations.

DLP Started Later (M6): DLP migration began later due to legal constraints and cross-team coordination requirements.

Parallel Operations: SOC continued on legacy stack (RSA, ESET) while new platform matured. Sentinel became primary only after gate criteria were met.

Migration Overview

The SIEM migration follows a methodical approach: discover what exists, plan what moves where, engineer the ingestion, enable detections, and transition the SOC. Throughout this process, both SIEMs remained operational until cutover criteria were met.

Data Residency & GDPR Compliance

Our approach depends on client requirements:

If client contract mandates logs stay in EU:

  • Deploy separate Sentinel workspace in EU region
  • Replicate detection logic through Git-based deployment model
  • Ensures data never leaves EU boundaries

If no strict residency requirement:

  • Centralize logging in primary workspace
  • Rely on Standard Contractual Clauses (SCCs) for cross-border transfers
  • RBAC, auditing, and retention controls to meet GDPR obligations

Key Decisions

Sentinel vs Splunk: Clear Separation

RevFlow already uses Splunk Cloud for operational analytics. Rather than migrate everything to Sentinel, we maintain clear ownership:

Microsoft Sentinel (Security)
‚úì Identity logs (Entra ID sign-in, audit)
‚úì Security alerts (MDE, Defender for Cloud)
‚úì Cloud control plane (Azure Activity, AWS CloudTrail)
‚úì Network security (Firewall, NSG flows)
‚úì Salesforce audit logs
‚úì Endpoint telemetry (MDE advanced hunting)
Splunk Cloud (Operations)
‚úì Application performance logs
‚úì Business transaction logs
‚úì Integration telemetry (API metrics)
‚úì Custom app metrics
‚úì SRE dashboards
‚úì Engineering troubleshooting
Why Not Consolidate Everything?

Engineering teams already have dashboards, alerts, and workflows built in Splunk. Migrating those would add 6+ months and significant risk. The clear separation also prevents ownership confusion‚ÄîSecurity owns Sentinel, Engineering owns Splunk.

Native Tables vs ASIM Normalization

We use native tables directly for 95% of queries. ASIM (Advanced Security Information Model) normalization is only used for custom application logs that need transformation.

Why native tables?

  • Microsoft's built-in detection content uses native tables
  • Better query performance (no function overhead)
  • Simpler troubleshooting
  • Most companies operate this way

Single Workspace vs Regional Workspaces

Despite serving clients in North America, Europe, and APAC, we deployed a single Sentinel workspace in Canada rather than regional workspaces.

Architecture Principle

"Global clients don't force regional SIEMs ‚Äî regulatory pressure and data sensitivity do."

We optimized for detection consistency and operational efficiency while meeting compliance through legal safeguards (SCCs) rather than infrastructure fragmentation.

Future-Ready: If customer contracts or regulators require EU data residency, we can introduce a regional EU workspace and federate detections without redesigning our security model.

Industry Best Practices

SIEM Implementation Best Practices

Gartner SOC Modernization:

  • XDR Integration: SIEM should integrate with XDR for unified detection and response
  • Cloud-Native First: Prioritize cloud-native SIEM for cloud-first organizations
  • Automation: Implement SOAR for playbook automation and response
  • Cost Optimization: Use tiered storage and filter high-volume, low-value logs

Microsoft Sentinel Best Practices:

  • Enable M365 Defender connector for free XDR telemetry
  • Use commitment tiers at 100+ GB/day for 50% savings
  • Implement workspace design based on RBAC, not geography (unless required)
  • Use Content Hub solutions before writing custom detections
  • Enable UEBA for entity-based threat detection
