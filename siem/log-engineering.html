Log Engineering

SIEM Track Technical Implementation

Log engineering is where logs are actually configured to flow into Sentinel. This involves Data Collection Rules (DCRs), ingestion-time transformations, cost tier decisions, and workspace architecture.

Workspace Architecture & Data Residency

Before configuring log ingestion, determine the workspace architecture based on compliance requirements.

Data Residency & GDPR Compliance

Our approach depends on client requirements:

If client contract mandates logs stay in EU:

  • Deploy a separate Sentinel workspace in an EU region (e.g., West Europe, North Europe)
  • Replicate detection logic through a Git-based deployment model
  • Analytics rules, workbooks, and playbooks stored in Git and deployed via CI/CD
  • Data never leaves EU boundaries

If no strict residency requirement:

  • Centralize logging in primary workspace
  • Rely on Standard Contractual Clauses (SCCs) for cross-border transfers
  • RBAC, auditing, and retention controls to meet GDPR obligations

Multi-Workspace Deployment Model

ScenarioWorkspace ArchitectureDetection Deployment
Single region, no residency requirements Single centralized workspace Direct deployment
EU data residency required Separate EU workspace Git-based CI/CD to both workspaces
Multi-region global deployment Regional workspaces + Azure Lighthouse Git-based deployment + cross-workspace queries

Windows Security Events DCR

Don't collect all Windows Security events—that's expensive and noisy. Use a DCR to filter to security-relevant events only.

Volume Reduction

Filtering Windows Security Events at ingestion can reduce volume by 60-80%. A server generating 10 GB/day of Security events might only need 2 GB of security-relevant events.

Ingestion-Time Transformations

DCRs can transform data using KQL before it reaches the workspace. This is useful for:

  • Dropping unnecessary columns to reduce storage
  • Parsing unstructured data into fields
  • Enriching data with static values
  • Filtering rows based on content

Example: Custom Application Log Transformation

RevFlow's application sends JSON security events. We transform them at ingestion to extract useful fields.

KQL (Transformation) 
// Transform RevFlow security logs at ingestion
source
| extend parsedJson = parse_json(RawData)
| extend 
    EventType = tostring(parsedJson.event_type),
    UserEmail = tostring(parsedJson.user.email),
    SourceIP = tostring(parsedJson.source_ip),
    Action = tostring(parsedJson.action),
    Resource = tostring(parsedJson.resource),
    Result = tostring(parsedJson.result),
    Timestamp = todatetime(parsedJson.timestamp)
| where EventType in ("login", "api_auth", "data_export", "permission_change")
| project-away RawData, parsedJson

Cost Tiers: Analytics vs Basic Logs

Sentinel offers two log tiers with different costs and capabilities:

Feature Analytics Logs Basic Logs
Cost (Pay-as-you-go) $2.76/GB $0.50/GB
Retention (included) 90 days 8 days
Query capability Full KQL Limited (basic operators only)
Analytics rules Yes No
Workbooks/Dashboards Yes Limited
Search jobs Yes Yes
Best for Detection, investigation, dashboards Compliance, forensics, high-volume/low-query

Basic Logs Candidates

Move high-volume, low-query data to Basic Logs:

Log Type Daily Volume Recommended Tier Reason
NSG Flow Logs ~50 GB Basic High volume, mostly for forensics
AWS VPC Flow Logs ~15 GB Basic High volume, mostly for forensics
Verbose firewall logs ~10 GB Basic Keep threat logs in Analytics, traffic logs in Basic
Sign-in logs (archive) Varies Basic Compliance retention beyond 90 days

Configure Basic Logs

Azure CLI 
# Set table to Basic Logs tier
az monitor log-analytics workspace table update \
  --resource-group "rg-sentinel" \
  --workspace-name "law-sentinel-prod" \
  --name "AzureNetworkAnalytics_CL" \
  --plan "Basic"

# Verify table plan
az monitor log-analytics workspace table show \
  --resource-group "rg-sentinel" \
  --workspace-name "law-sentinel-prod" \
  --name "AzureNetworkAnalytics_CL" \
  --query "plan"

Workspace Architecture

For RevFlow, a single workspace with regional data residency is sufficient:

Design Decision Choice Rationale
Number of workspaces Single workspace Simplicity, unified detection, single pane of glass
Region East US 2 Primary Azure region, data residency compliance
Commitment tier 100 GB/day ~30% discount at this volume
Retention 90 days interactive, 2 years archive SOC 2 compliance requirement

Workspace Configuration

Azure CLI 
# Create Log Analytics Workspace
az monitor log-analytics workspace create \
  --resource-group "rg-sentinel" \
  --workspace-name "law-sentinel-prod" \
  --location "eastus2" \
  --sku "PerGB2018" \
  --retention-time 90

# Enable Sentinel on the workspace
az sentinel onboard create \
  --resource-group "rg-sentinel" \
  --workspace-name "law-sentinel-prod"

# Set commitment tier (100 GB/day)
az monitor log-analytics workspace update \
  --resource-group "rg-sentinel" \
  --workspace-name "law-sentinel-prod" \
  --capacity-reservation-level 100

Validation Queries

Use these queries to validate data is flowing correctly:

KQL 
// Check ingestion volume by table (last 24h)
Usage
| where TimeGenerated > ago(24h)
| summarize TotalGB = sum(Quantity) / 1024 by DataType
| order by TotalGB desc

// Check for ingestion delays
SigninLogs
| where TimeGenerated > ago(1h)
| extend IngestionDelay = ingestion_time() - TimeGenerated
| summarize 
    AvgDelayMinutes = avg(IngestionDelay) / 1m,
    MaxDelayMinutes = max(IngestionDelay) / 1m,
    Count = count()

// Verify specific connector is working
AzureActivity
| where TimeGenerated > ago(1h)
| summarize count() by bin(TimeGenerated, 5m)
| render timechart

// Check DCR-collected Windows events
SecurityEvent
| where TimeGenerated > ago(1h)
| summarize count() by EventID
| order by count_ desc
