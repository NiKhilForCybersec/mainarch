Discovery & Assessment

SIEM Track Weeks 1-6

Before migrating anything, you must understand what exists. This phase involves extracting RSA NetWitness configurations, meeting with stakeholders to identify log sources, and documenting the current state. Expect this to take 4-6 weeks.

The Reality of Discovery

Discovery is harder than it sounds. You can't just "ask for a list of log sources" — that list doesn't exist in most organizations, and even if it does, it's outdated. Here's the practical approach:

Expect Documentation to Be Wrong

Assume 50% of existing documentation is outdated. Trust actual configurations over documentation. Validate everything with real traffic analysis.

Step 1: RSA NetWitness Export Analysis

Start with what RSA actually has configured. Export these items from the RSA NetWitness console:

Log Source Configuration Export

Export Item Location in RSA What You Learn
Log Decoder sources Admin → Services → Log Decoder → Sources All configured log sources by type
Event Source Management Admin → Event Sources → Manage Source names, IPs, collection methods
Parser configurations Admin → Services → Log Decoder → Parsers Custom parsers that may need recreation
ESA rules Configure → ESA Rules Detection rules to translate
Reporting rules Configure → Reports Compliance reports to recreate

Volume Analysis

Pull 30-day ingestion statistics per source. This is critical for Sentinel cost planning.

RSA Query 
// RSA NetWitness - Get volume by device type (last 30 days)
// Run in Investigation > Navigate view

device.type exists
| group by device.type
| sort by total desc

Document results in a spreadsheet with columns:

  • Source Name
  • Source IP/Hostname
  • Device Type (RSA classification)
  • Daily Volume (GB)
  • Monthly Volume (GB)
  • Last Event Timestamp (is it still active?)
  • Owner Team
  • Business Criticality

Identify Dead Sources

Sources configured but not sending data are common. Don't migrate dead sources.

RSA Query 
// Find sources with no events in last 7 days
// Compare against configured sources list

device.ip exists
| where time > 7 days ago
| group by device.ip
| sort by total asc

Step 2: Network Discovery

RSA doesn't know about everything. Some sources might send logs via syslog that RSA receives but doesn't properly classify. Work with the network team to identify all log-sending infrastructure.

Identify Syslog Senders

Bash 
# On firewall/network device - find what's sending to RSA
# Replace RSA_COLLECTOR_IP with actual IP

# Check firewall rules allowing syslog to RSA
show access-list | include 514
show access-list | include 6514

# Or capture traffic
tcpdump -i any port 514 -c 1000 | awk '{print $3}' | sort | uniq -c | sort -rn

Review Load Balancer Configurations

If RSA uses a load balancer for log collection, review the VIP configurations to identify all source networks.

Step 3: Stakeholder Collaboration

You need information from teams across the organization. Here's the realistic expectation for each stakeholder:

Stakeholder What They'll Tell You What You Actually Need to Discover Questions to Ask
IT Operations "Everything is documented in Confluence" Confluence is 3 years stale; get actual inventory Can I get CMDB export? What monitoring tools do you use?
Application Owners "We send logs to Splunk" They also send security events to NetWitness via legacy forwarder Where does your app log to? Any security-specific logging?
Network Team "Here's our firewall list" 4 shadow firewalls exist nobody mentioned Show me all devices with syslog config. Any lab/dev firewalls?
Database Team "We have audit logging enabled" Only on 2 of 15 databases Which databases have audit logging? Where do those logs go?
Identity Team "We use AD and Entra ID" 12 service accounts with Domain Admin nobody tracks Where are AD security logs forwarded? What about Entra ID?
Cloud Team "Azure and some AWS" 3 AWS accounts, 2 Azure subscriptions, no central logging List all cloud accounts. Is CloudTrail/Activity Log enabled?
Security Team "All our detections are critical" 80% of rules haven't triggered in 6 months Which rules fire most? Which are compliance-required?
Salesforce Admin "We have Event Monitoring enabled" Shield Event Monitoring not purchased, basic logs only What Salesforce edition? Is Shield Event Monitoring licensed?

Intake Meeting Template

For each team, schedule a 30-minute intake meeting. Come prepared with specific questions, not open-ended "tell me about your logs."

Pro Tip

Bring screenshots from RSA NetWitness showing logs from their systems. "I see we're getting Windows Security logs from SERVER01 — is that still active?" gets better answers than "What servers do you have?"

Meeting Agenda Template

Markdown 
## Log Source Discovery - [Team Name]
**Date:** [Date]
**Attendees:** [Names]

### Current State Questions

Step 4: CMDB and Asset Inventory

Request exports from Configuration Management Database (CMDB) and compare against RSA sources:

CMDB Category Expected Log Sources Check RSA For
Windows Servers Security Event Log, PowerShell, Sysmon (if installed) device.type = 'winevent_nic'
Linux Servers auth.log, syslog, application logs device.type = 'syslog'
Network Devices Firewall logs, switch logs, VPN logs device.type contains 'firewall' or 'cisco'
Databases Audit logs, error logs device.type = 'database'
Web Servers Access logs, error logs, WAF logs device.type = 'apache' or 'iis'

Step 5: Cloud Resource Discovery

Azure Resources

Azure CLI 
# List all subscriptions
az account list --output table

# For each subscription, list resources that should be logging
az resource list --subscription "Sub-Name" --output table

# Check if diagnostic settings are enabled
az monitor diagnostic-settings list --resource "/subscriptions/{sub-id}/resourceGroups/{rg}/providers/{provider}/{resource}"

AWS Resources

AWS CLI 
# Check CloudTrail status
aws cloudtrail describe-trails

# Check if CloudTrail is logging
aws cloudtrail get-trail-status --name default

# List S3 buckets (potential log destinations)
aws s3 ls

# Check GuardDuty status
aws guardduty list-detectors

Salesforce Audit Configuration

For RevFlow's Salesforce-centric business, understanding Salesforce audit logging is critical:

Salesforce Feature What It Logs License Required
Setup Audit Trail Admin configuration changes Included (all editions)
Login History User logins (last 6 months) Included (all editions)
Field History Tracking Field-level changes (configured objects) Included (limited fields)
Event Monitoring API calls, report exports, logins Shield add-on ($$$)
Real-Time Event Monitoring Streaming events for SIEM integration Shield add-on ($$$)
Salesforce Shield Licensing

If RevFlow doesn't have Salesforce Shield Event Monitoring, the audit data available for SIEM ingestion is limited. Verify licensing with the Salesforce admin before planning detections.

Step 6: Document Current RSA Detection Rules

Export all ESA (Event Stream Analysis) rules from RSA NetWitness. Categorize them for migration:

Category % of Rules (Typical) Action
Never fired in 12 months 30-40% Discard
Fires constantly (noisy) 20-30% Review/Fix or Discard
Duplicated by Microsoft content 20% Use Microsoft version
Actually valuable custom 10-20% Migrate and improve
Compliance-required 5-10% Must migrate

Discovery Deliverables

At the end of discovery, you should have:

  • Log Source Inventory Spreadsheet — All sources with volume, owner, and migration decision
  • RSA ESA Rule Export — All detection rules with categorization
  • Stakeholder Contact List — Who owns what, for future questions
  • Gap Analysis — Sources that should be logging but aren't
  • Volume Estimates — Expected Sentinel ingestion cost projection
  • Dependency Map — What relies on RSA today (dashboards, reports, integrations)
Discovery Complete Checklist
  • ☐ RSA log source configuration exported
  • ☐ 30-day volume analysis complete
  • ☐ All stakeholder intake meetings complete
  • ☐ CMDB/asset inventory compared against RSA sources
  • ☐ Cloud resources documented (Azure, AWS)
  • ☐ Salesforce audit capabilities confirmed
  • ☐ ESA rules exported and categorized
  • ☐ Log source inventory spreadsheet finalized
  • ☐ Sentinel cost projection calculated
