2. The 6-Disposition Framework

Every RSA rule was assigned one of six dispositions. This prevents "convert everything" paralysis and ensures decisions are justified and auditable.

RETIRE â†’ XDR

XDR detects this technique natively with superior fidelity. Do not migrate â€” document XDR coverage.

REPLACE â†’ Content Hub

Sentinel Content Hub has equivalent rule. Enable built-in, tune thresholds, no custom KQL needed.

MIGRATE â†’ Custom KQL

No XDR or Content Hub coverage. Requires custom Sentinel analytics rule development.

DEFER

Blocked by missing telemetry. Track dependency, enable when data source is onboarded.

KEEP TEMP

Run in RSA during parallel operations. Maintain until Sentinel coverage is validated.

DROP

No value â€” noisy, stale IOCs, or obsolete technology. Migration cleans house.

3. Migration Methodology

Step-by-step process applied to each RSA rule. Items I owned or contributed to are marked.

1
Export & Normalize Intent

Extract RSA rule details (name, logic, data sources, severity). Write single "intent" sentence describing what behavior the rule detects.

2
Map to MITRE ATT&CK + Business Objective

Identify technique ID and business risk category. Rules that can't map to MITRE are candidates for DROP.

3
Check Telemetry Availability My Role

Validate: Is the required data source onboarded? Is it normalized? Is data quality acceptable? Mark as DEFER if blocked.

4
Check XDR Native Coverage My Role

Does MDE/MDI/MDO/MDA detect this technique? Evidence required: Microsoft docs, historical alerts, or simulation test.

5
Check Content Hub & Enable Built-ins My Role

Does Sentinel Content Hub have equivalent? If yes, enable, tune thresholds, configure entity mapping â€” no custom KQL needed.

6
Assign Disposition & Document

Record final decision with evidence type, justification, owner, and date. This is the audit trail for compliance.

Evidence Types for Retirement Decisions

DOC Microsoft documentation confirms XDR coverage   TEST Simulation/purple team validated detection fired   HIST Historical incidents show XDR detected reliably   N/A Not applicable (DROP or Content Hub)

Why This Approach Works

Outcome-driven, not rule-driven. We didn't translate rules 1:1. We evaluated whether the attack technique was detected â€” not whether the rule logic was preserved.

Ingestion follows ownership. We decided what platform owns each detection FIRST, then ingested only the logs needed. This reduced costs by 60-80% compared to "ingest everything" approaches.

Built-in before custom. Every custom KQL rule is technical debt. We used Content Hub wherever possible â€” 102 of 255 migrated Sentinel rules were built-ins, not custom.

Modernization, not migration. The 230 new rules are the real value â€” we didn't just move rules, we built coverage for a modern threat landscape RSA couldn't address.
