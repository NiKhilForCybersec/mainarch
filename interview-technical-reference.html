Key Interview Insight

When asked "what goes to Sentinel?" — Answer with this framework:

  1. What XDR can't SEE: Network, non-Microsoft SaaS, on-prem apps
  2. What XDR can't KNOW: HR context (termination), business logic (change windows), role-based policies
  3. What XDR can't CORRELATE: Cross-domain sequences spanning 3+ data sources

If it's a pure endpoint/identity/email attack → XDR. If it needs context or correlation → Sentinel.

2. Rapid-Fire Q&A 10-20 sec answers

Memorize these. Deliver confidently. Short, punchy, impressive.

SIEM Design & Detection Strategy

Question10-20 Second Answer
"What's detection-driven migration?" "Instead of porting every legacy rule 1-to-1, we evaluate detection intent. XDR handles endpoint, identity, email natively — suspicious PowerShell, LSASS dumps, phishing — those get retired. Sentinel handles what XDR can't see — terminated employee data access, cross-domain correlation, business logic. Result: 780 rules retired to XDR, 485 purpose-built Sentinel rules."
"Why XDR-first?" "XDR has kernel-level visibility, AMSI integration, and ML-based detection that SIEM queries can't match. Example: LSASS credential dumping — MDE sees the memory access attempt before any log is written. PowerShell — AMSI shows decoded script content. Phishing — MDO detonates attachments in sandbox. You configure controls, you don't write malware rules. Microsoft owns detection logic, we own trust decisions."
"Sentinel vs Splunk difference?" "Splunk is SIEM-first — you ingest everything, build all detections yourself. Microsoft is XDR-first — correlation happens upstream, Sentinel fills gaps. It's not feature comparison, it's architectural difference."
"What does Splunk ES add?" "ES is the security brain on top of Splunk Enterprise. Adds prebuilt correlation searches, CIM enforcement, risk-based alerting, SOC workflows, and threat intel framework. You're paying for content maturity."
"Single workspace or multiple?" "Single workspace, multi-region DCRs for data residency. Simpler RBAC, unified correlation, easier management. Complexity is the enemy of security."
"How do you handle cost?" "Detection-driven ingestion — don't ingest logs you won't detect on. Storage tiers for compliance data. Commitment tiers for 50% savings. E5 grants 5MB/user/day free. Design for cost from Day 1."

Log Source Onboarding

Question10-20 Second Answer
"How do you prioritize log sources?" "What XDR doesn't cover comes first. Identity control-plane logs — Entra AuditLogs — because XDR detects brute force and impossible travel, but NOT who got added to Global Admins. Then XDR alerts for correlation. Then network logs XDR can't see — firewalls, proxies, VPN. Then SaaS apps outside Microsoft — Salesforce, Workday audit logs."
"Why not ingest MDE telemetry to Sentinel?" "50 million events per day, $15K/month, and XDR already has it. We ingest SecurityAlert and SecurityIncident — high value, low volume, often free. Query raw telemetry in XDR Advanced Hunting."
"Custom log onboarding process?" "DCR with transformation at ingestion, not query time. Validate field parsing, timestamp accuracy, expected volume. Then align detections. Bad data equals bad detections."
"Sources that can't run agents?" "Defender for IoT for passive monitoring. Syslog/CEF for network devices. API connectors for SaaS. Network segmentation for unmanaged. Every device: agent, logs, or network monitoring."

Detection Engineering & MITRE

Question10-20 Second Answer
"Built-in vs custom rules?" "Content Hub first — 102 built-in rules enabled for commodity threats like known malware, brute force, suspicious sign-ins. Custom KQL only for gaps — 58 rules for business-specific: terminated employee correlation, service account misuse, SAP transactions outside change windows, VPN + file access patterns. Don't reinvent what Microsoft maintains."
"How do you map to MITRE?" "Every detection tagged with technique ID. ATT&CK blade in Sentinel for visualization. Focus on techniques attackers actually use against our industry, not checkbox completeness."
"Detection validation approach?" "ATT&CK coverage mapping, SOC feedback loop, real incident analysis, and purple team exercises. Validate coverage, not just rule count."
"How do you reduce false positives?" "Audit mode first, always. Build known-good baseline. Document every exclusion with business justification. Periodic review. False positives are expected — plan for them."

Windows, Network & Protocols

Question10-20 Second Answer
"Critical Windows events?" "4624/4625 for logon, 4688 for process creation with command line, 4720 for user creation, 7045 for service installation, 1102 for log clearing. But MDE captures this better than event logs."
"Lateral movement detection?" "For MDE-covered endpoints, XDR handles it — SMB abuse, RDP pivoting, WMI execution, PsExec patterns. MDE sees the actual process chain and behavioral intent. Where Sentinel adds value: lateral movement involving non-MDE systems, or correlation with network logs showing the actual traffic flows. Also, lateral movement BY a compromised account flagged in identity signals."
"Network security monitoring?" "Firewall logs to Sentinel, Defender for IoT for OT/IoT passive monitoring, NetFlow for traffic patterns. Segmentation isolates what we can't instrument. Layered visibility."
"Sysmon vs MDE?" "MDE for everything it covers — kernel-level telemetry, behavioral detection, cloud-backed ML. Sysmon logs process creation, but MDE understands intent — is this PowerShell malicious? Sysmon gives you event ID 1; MDE gives you 'Suspicious credential access.' Sysmon only for legacy systems MDE can't protect — Server 2008, old Linux. For modern systems, MDE is Sysmon on steroids."

Deployment & Operations

Question10-20 Second Answer
"ASR deployment strategy?" "Audit, Warn, Block — in that order. 4-8 weeks per wave. Build known-good baseline before enforcement. Never disrupt legitimate business workflows. XDR owns detection, we own trust decisions."
"EDR coexistence approach?" "Active-active, not passive. Both ESET and MDE protecting simultaneously. Real detection comparison data. Zero protection gap at any point. Remove legacy only after new is proven."
"When do you cutover?" "Gate-based, not calendar-based. 70-80% endpoint coverage, policy compliance, detection parity confirmed, 90 days stable telemetry. Cutover when ready, not when planned."
"SOC transition approach?" "Parallel operations, both SIEMs running. KQL training before switch. Unified queue in M365 Defender as primary. Legacy stays until new is proven."
"How do you handle incidents?" "Investigate in XDR — that's where raw telemetry lives. Correlate in Sentinel for broader context. M365 Defender unified queue is the SOC's home, not Sentinel."

Scripting & Automation

Question10-20 Second Answer
"KQL vs SPL?" "SPL is pipe-based like Unix, KQL is more SQL-like. Different syntax, same concepts. I work primarily in KQL now, but the detection logic transfers — time windows, aggregations, joins, thresholds."
"How do you automate?" "Logic Apps for SOAR playbooks — auto-enrichment, ticket creation, containment actions. PowerShell for admin tasks. Python for data processing. Automate repetitive, keep humans for judgment."
"Favorite KQL technique?" "Lookback joins with ago() and join kind=inner. Find event A, then look for event B in a time window. That's where correlation becomes powerful — single events aren't interesting, sequences are."

3. Detailed Answers 60-90 sec

For "walk me through" questions. Shows depth and experience.

"Walk me through your SIEM migration approach"
Detailed Response (60-90 sec)
"Our migration was detection-driven, not log-first.

PHASE 1 — DISCOVERY: Documented 1,247 RSA rules. But instead of translating them 1-to-1, we normalized them into security outcomes — credential abuse, lateral movement, data exfiltration.

PHASE 2 — OWNERSHIP: For each outcome, we asked: does XDR handle this natively?

Examples we RETIRED to XDR (780 rules):
• Malicious PowerShell patterns → MDE has AMSI, sees decoded scripts
• LSASS memory access → MDE kernel-level credential guard integration
• Suspicious email attachments → MDO sandbox detonation
• Brute force authentication → MDI + Entra ID Protection
• Lateral movement via SMB/RDP → MDE behavioral detection
• Known malware hashes → MDE cloud protection, real-time updates

Examples we KEPT/BUILT in Sentinel (485 rules):
• Terminated employee accessing sensitive data → HR feed + SharePoint logs
• VPN from impossible location + file download → Network + identity correlation
• Service account interactive login → Not an "attack," but policy violation
• Firewall blocked → allowed → blocked pattern → Network behavioral
• OAuth app consent from risky user → MCAS + identity risk score
• Off-hours SAP transaction → Business app + time-based logic

PHASE 3 — SENTINEL FOCUS: 485 rules for what Sentinel does best — cross-domain correlation, non-Microsoft telemetry, business logic. 255 migrated from RSA, plus 230 NEW rules RSA never had — SaaS coverage, compliance, XDR correlation.

PHASE 4 — PARALLEL OPS: Both SIEMs running for 90+ days. Cutover only after gate criteria met.

Result: 12-month timeline instead of 18, reduced ingestion cost, XDR detecting from Day 1."
"How do you handle systems where you can't install MDE?"
Detailed Response (60-90 sec)
"The goal is 100% visibility, not 100% agent deployment.

For each asset class, we implement the right compensating control:

NETWORK DEVICES — Firewalls, switches can't run agents. Syslog/CEF to Sentinel. NetFlow for traffic analysis.

OT/IoT — Building management, cameras, printers. Defender for IoT provides passive network monitoring. No agent needed — analyzes traffic for anomalies.

LEGACY OS — Server 2008, old Linux. Sysmon plus WEF for Windows, Auditd or Wazuh for Linux. Network segmentation limits blast radius.

VENDOR-MANAGED — ERP systems, trading terminals. Require SOC 2, API log integration where possible, strict segmentation.

SAAS — No endpoint at all. Defender for Cloud Apps as CASB. API connectors for audit logs.

Every device either has an agent, sends logs, or is monitored passively on the network. That's 100% visibility."
"Explain your detection engineering methodology"
Detailed Response (60-90 sec)
"I follow a 5-phase outcome-driven approach:

PHASE 1 — NORMALIZE TO OUTCOMES: Legacy SIEM tells you what behaviors mattered, not what rules to copy. 'Credential abuse' is an outcome. 'Rule #47: LSASS access' is implementation.

PHASE 2 — DECIDE OWNERSHIP: For each outcome, who handles it better? Let me give you concrete examples:

XDR HANDLES (Retire from SIEM):
• Suspicious PowerShell execution → MDE has AMSI integration, sees script content
• LSASS credential dumping → MDE has kernel-level visibility, behavioral detection
• Phishing email with malicious attachment → MDO detonates in sandbox
• Impossible travel sign-in → MDI + Entra ID Protection, ML-based
• Pass-the-Hash → MDI detects authentication anomalies
• Ransomware encryption behavior → MDE behavioral blocking

SENTINEL HANDLES (Build/Migrate):
• VPN login from terminated employee → Requires HR feed + identity correlation
• Bulk SharePoint download after resignation notice → Business context needed
• Firewall allow after multiple blocks from same IP → Network + threat intel
• Service account used interactively → Identity governance rule
• Data exfil to personal OneDrive → MCAS + DLP + user risk correlation
• SAP privileged transaction outside change window → Business app + time logic
• Same user: MFA reset + password change + new device + sensitive file access within 1 hour → Higher-order correlation across 4 data sources

PHASE 3 — DEFINE INGESTION: Ingestion follows detection, not the other way around. Since XDR owns PowerShell, LSASS, malware detection — I don't ingest DeviceProcessEvents to Sentinel. That's 50M events/day we don't pay for.

PHASE 4 — DESIGN DETECTIONS: Content Hub first for commodity threats. Custom KQL for business-specific logic. Higher-order correlation that combines XDR alerts with network and SaaS signals.

PHASE 5 — VALIDATE: ATT&CK mapping, SOC feedback, real incidents, purple team. Coverage validation, not rule count.

This methodology compressed our migration from 18 months to 12 and gave us better coverage than the legacy SIEM ever had."
"How do you optimize Sentinel costs?"
Detailed Response (60-90 sec)
"Cost optimization starts at architecture, not after deployment.

PRINCIPLE 1 — DETECTION-DRIVEN INGESTION: Every log source must justify itself with specific detection use cases. No 'ingest everything just in case.'

PRINCIPLE 2 — XDR INDEPENDENCE: MDE telemetry goes to Defender backend directly. XDR works without Sentinel. We ingest alerts, not 50 million raw events per day.

PRINCIPLE 3 — STORAGE TIERS: Analytics tier for active detections. Basic tier for search-only compliance logs — 60% cheaper. Archive for long-term retention at lowest cost.

PRINCIPLE 4 — COMMITMENT TIERS: Design for commitment from start. 100GB/day commitment saves 50% vs pay-as-you-go. E5 license grants 5MB/user/day free — that's significant.

PRINCIPLE 5 — TRANSFORMATION AT INGESTION: Filter and transform in DCRs before data hits the workspace. Don't pay to store fields you'll never query.

Result: Sentinel costs about $12K/month for our environment instead of the $50K it would cost with naive ingestion."

4. Technical Scenarios

Real-world scenarios a Security Consulting Manager would ask to test practical knowledge.

Detection Engineering Scenarios

"You discover a gap in MITRE ATT&CK coverage for T1059 (Command and Scripting Interpreter). Walk me through your approach."
Quick (20 sec)

"First, check what XDR already covers — MDE detects malicious PowerShell, encoded commands, AMSI bypasses natively. If XDR handles it, document and move on. If there's a genuine gap — maybe abuse of a business-specific script or interpreter XDR doesn't monitor — then build custom KQL."

Detailed (60 sec)
"T1059 is broad — covers PowerShell, CMD, VBScript, Python, etc. First, I check XDR coverage:

T1059 SUB-TECHNIQUES XDR HANDLES (Don't build in Sentinel):
• T1059.001 PowerShell → MDE AMSI integration, sees decoded content
• T1059.003 Windows CMD → MDE behavioral, parent-child analysis
• T1059.005 VBScript → MDE script block logging integration
• T1059.007 JavaScript → MDE + MDO for email-based delivery
• Encoded commands (-enc) → MDE decodes and analyzes
• Download cradles (IEX, WebClient) → MDE network + process correlation

WHAT MIGHT NEED SENTINEL:
• Business-specific script interpreter XDR doesn't know
• Python scripts on servers (if not MDE-covered Linux)
• Script execution + network connection to rare domain (correlation)
• Script execution by terminated employee (business context)

IF CUSTOM RULE NEEDED:
• Check data availability: 4688 with command line, or DeviceProcessEvents
• Build detection: Encoded commands, download patterns, unusual parent-child
• Tune: Baseline legitimate admin scripts, exclude by path or signer

But honestly, 90% of T1059 is XDR territory. I'd only build custom if there's a specific gap XDR doesn't cover — and I'd document why XDR doesn't cover it."
"A detection rule is generating 500 alerts per day. How do you tune it?"
Quick (20 sec)

"First, analyze the alerts — are they true positives being ignored, or false positives? If false positives, identify the pattern causing noise — usually a specific user, host, or application. Add targeted exclusions with documented business justification, or adjust thresholds. Never disable without understanding why it's firing."

Detailed (60 sec)
"500 alerts/day means either the rule is broken or we have a real problem. Systematic approach:

STEP 1 — SAMPLE ANALYSIS: Pull 50-100 alerts, manually triage. What percentage are true positives vs false positives?

STEP 2 — IF TRUE POSITIVES: 500 real attacks per day is a bigger problem. Escalate, investigate root cause, maybe we have a compromised system or misconfiguration driving the activity.

STEP 3 — IF FALSE POSITIVES: Identify the pattern. Usually it's:
• Specific service account doing legitimate automation
• IT admin tool triggering detection
• Scheduled task or monitoring software
• One misconfigured system

STEP 4 — TARGETED EXCLUSION: Don't disable the rule. Add specific exclusion for the known-good activity. Document the business justification — what tool, what purpose, who approved.

STEP 5 — THRESHOLD ADJUSTMENT: If it's a volumetric rule, maybe the threshold is too low for our environment. Baseline normal and set threshold above that.

STEP 6 — VALIDATE: After tuning, monitor for a week. Did noise drop? Are we still catching real threats?

The goal is fewer, higher-fidelity alerts — not zero alerts."
"You need to detect insider threat — specifically a terminated employee accessing data before their last day. How would you approach this?"
Quick (20 sec)

"This is HR integration plus detection. Workday or HR system sends termination notice, we flag the user in a watchlist. Then correlate: bulk file downloads, SharePoint access spikes, email forwarding to personal accounts, USB activity via MDE device control. Trigger alert when terminated user exhibits any of these behaviors."

Detailed (60 sec)
"Insider threat detection requires business context that pure security telemetry doesn't have.

DATA SOURCES NEEDED:
• HR feed: Termination date, notice period, department
• Identity: Sign-in patterns, access times
• File activity: SharePoint, OneDrive, file servers
• Email: Forwarding rules, external sends, attachment volume
• Endpoint: USB usage, cloud storage uploads, printing

DETECTION LOGIC:
1. HR feed flags user as 'termination pending'
2. User added to elevated monitoring watchlist
3. Correlate against suspicious behaviors:
   • File downloads > 100 in 24 hours
   • Access outside normal hours
   • New email forwarding rules
   • USB device insertion (if not normally used)
   • Cloud storage upload (Dropbox, personal OneDrive)

IMPLEMENTATION:
• Watchlist in Sentinel with user identifiers
• Analytics rule: Join HR feed with file/email activity
• Higher sensitivity thresholds for watchlist users
• Alert goes to HR and Security, not just SOC

KEY INSIGHT: This is a business process detection, not just technical. Need HR partnership and legal sign-off on monitoring."
"A new zero-day is announced (let's say Log4Shell). How do you quickly assess exposure and implement detection?"
Quick (20 sec)

"Three parallel tracks: First, asset inventory — what runs Java/Log4j? TVM in MDE helps here. Second, detection — implement known IOCs, JNDI lookup patterns in network logs, exploitation attempts in WAF. Third, hunting — proactive search for evidence of prior exploitation. All within first few hours."

Detailed (60 sec)
"Zero-day response is a race. I'd run three parallel workstreams:

TRACK 1 — EXPOSURE ASSESSMENT (First 2 hours):
• Query TVM in MDE for Log4j vulnerabilities
• Asset inventory: What runs Java? Application teams confirm
• Internet-facing vs internal exposure
• Prioritize: Internet-facing, critical systems first

TRACK 2 — DETECTION IMPLEMENTATION (First 4 hours):
• IOCs from threat intel (IPs, domains, hashes) into TI feed
• Network detection: JNDI lookup patterns in firewall/proxy logs
  ${jndi:ldap://}, ${jndi:rmi://}
• WAF rules if available
• Endpoint: Suspicious Java child processes, outbound LDAP

TRACK 3 — THREAT HUNTING (First 24 hours):
• Historical search: Did we see exploitation attempts before the CVE was public?
• Network logs for JNDI patterns in past 30 days
• Endpoint: Java spawning cmd/PowerShell/bash unexpectedly

TRACK 4 — COMMUNICATION:
• Status updates to leadership every 2 hours
• Document exposure, detection coverage, gaps

Within 24 hours: Exposure mapped, detection in place, hunting complete, patch prioritization delivered."

Log Source & Ingestion Scenarios

"A critical business application generates 500GB of logs per day. The team wants full visibility. How do you handle it?"
Quick (20 sec)

"500GB at Sentinel rates is $1,000+ per day — unsustainable. First question: what detections require this data? Filter at source to security-relevant events only. Use Basic tier if needed for search without detection. Transform and summarize where possible. Maybe 500GB becomes 50GB of actual security value."

Detailed (60 sec)
"500GB/day is a cost and architecture problem, not just ingestion.

STEP 1 — QUESTION THE REQUIREMENT:
'Full visibility' is rarely needed. What specific threats or use cases require this data? Usually it's:
• Authentication events
• Errors and failures
• Admin actions
• Specific transaction types

STEP 2 — FILTER AT SOURCE:
If the app generates 500GB but only auth events matter, filter in the DCR transformation. Maybe 500GB becomes 20GB.

STEP 3 — TIERING STRATEGY:
• Analytics tier: Only what has active detection rules
• Basic tier: Search-only for troubleshooting, 60% cheaper
• Don't ingest: Debug logs, performance metrics, verbose traces

STEP 4 — SAMPLING:
For some use cases, 10% sampling is statistically valid. Anomaly detection can work on sampled data.

STEP 5 — ALTERNATIVE STORAGE:
Maybe this belongs in Splunk (if SRE owns it) or Azure Data Explorer for cheap long-term retention with occasional query.

STEP 6 — BUSINESS CONVERSATION:
'Full visibility costs $365K/year. Targeted security visibility costs $36K/year. What threats justify 10x cost?'

Usually, when you break down the actual detection requirements, 500GB becomes 50GB or less."
"Network team wants to onboard Palo Alto firewall logs. Walk me through the process."
Quick (20 sec)

"First, confirm log format — CEF over syslog is standard for Palo Alto. Set up AMA with syslog collection, configure firewall to send logs. Build DCR for parsing if needed. Validate data quality — fields parsing, timestamps. Then enable built-in Palo Alto analytics rules from Content Hub and tune for environment."

Detailed (60 sec)
"Palo Alto is a well-supported source, so mostly standard process:

STEP 1 — REQUIREMENTS GATHERING:
• Which log types? Traffic, Threat, URL, WildFire?
• Volume estimate? Plan for DCR capacity
• Retention requirements? Compliance drivers?
• What detections will use this data?

STEP 2 — ARCHITECTURE:
• Syslog forwarding to Azure Monitor Agent
• CEF format preferred (structured)
• Log Analytics workspace destination
• DCR for any transformations

STEP 3 — FIREWALL CONFIGURATION:
• Syslog server profile pointing to AMA collector
• Select log types to forward
• Severity filtering if needed (reduce noise)

STEP 4 — VALIDATION:
• Confirm logs arriving in CommonSecurityLog table
• Check field parsing: SourceIP, DestIP, Action, etc.
• Timestamp accuracy — critical for correlation
• Volume matches expectations

STEP 5 — DETECTION ENABLEMENT:
• Content Hub: Palo Alto solution pack
• Built-in rules for known threats
• Custom rules for our specific use cases:
  - Blocked-then-allowed patterns
  - Geographic anomalies
  - C2 beacon detection

STEP 6 — DOCUMENTATION:
• Log source in inventory
• DCR configuration documented
• Detection rules mapped to this source"
"You're getting duplicate events in Sentinel from the same source. How do you troubleshoot?"
Quick (20 sec)

"Duplicates usually mean multiple collection paths. Check if both legacy Log Analytics agent and AMA are running. Or multiple DCRs targeting same source. Or the source itself is sending to multiple destinations. Query by _ResourceId and TimeGenerated to identify the duplicate pattern, then eliminate the redundant path."

Detailed (60 sec)
"Duplicates waste money and break detection logic. Systematic debug:

STEP 1 — CONFIRM IT'S ACTUALLY DUPLICATES:
Query the table, look for identical events:
| summarize count() by TimeGenerated, Computer, EventID
| where count_ > 1

Are they exact duplicates or similar events?

STEP 2 — IDENTIFY THE SOURCE:
• Check _ResourceId field — different resource IDs = multiple paths
• Check Computer and source fields
• Look at ingestion timestamps (_TimeReceived) vs event time

STEP 3 — COMMON CAUSES:
• Legacy agent AND AMA running simultaneously
• Multiple DCRs collecting same data
• Source configured to send to multiple destinations
• Forwarding chain with multiple hops

STEP 4 — RESOLUTION:
If dual agents: Remove legacy agent, migrate to AMA only
If multiple DCRs: Consolidate into single DCR
If source misconfiguration: Fix at source

STEP 5 — VALIDATION:
After fix, monitor for 24 hours. Duplicates should stop.
Check detection rules didn't break from the change.

STEP 6 — COST RECOVERY:
If duplicates ran for a while, calculate wasted spend. Document for future architecture decisions."

Investigation & Response Scenarios

"You receive an alert for suspicious PowerShell execution on a finance workstation. Walk me through your investigation."
Quick (20 sec)

"Start in XDR, not Sentinel — MDE has the process tree. Look at parent process, command line, what spawned it. Check for encoded commands, network connections, file writes. Pivot to user — is this their normal behavior? Check timeline before and after. Scope — did this happen on other machines? Contain if malicious, document if false positive."

Detailed (60 sec)
"PowerShell alert on finance workstation — high priority. I investigate in XDR first because that's where telemetry lives.

PHASE 1 — INITIAL TRIAGE (5 minutes):
• Open alert in M365 Defender
• Review command line — is it obviously malicious?
  - Encoded (-enc)? Download cradle (IEX)?
• Parent process — what spawned PowerShell?
  - Explorer.exe (user) vs WMI vs scheduled task
• User context — finance user running PowerShell unusual?

PHASE 2 — PIVOT & SCOPE (10 minutes):
• Device timeline: What happened before and after?
• Network connections from this process
• Files written or modified
• Check other devices: Same command seen elsewhere?
• User activity: Login anomalies? Other alerts?

PHASE 3 — CONTEXT FROM SENTINEL (5 minutes):
• Same user: Recent sign-in anomalies?
• Same device: Other security events?
• Cross-reference with email: Phishing received?

PHASE 4 — DECISION:
If malicious: Isolate device via MDE, reset user credentials, block indicators, full IR process
If false positive: Identify root cause (IT tool, admin script), document and add exclusion, close alert with disposition

PHASE 5 — DOCUMENTATION:
Timeline, findings, actions taken, IOCs if any."
"You're seeing beaconing behavior in network logs — periodic connections to an external IP every 30 seconds. How do you investigate?"
Quick (20 sec)

"Beaconing is classic C2. First, identify the process making connections — correlate network logs with endpoint telemetry. Check the destination IP/domain against threat intel. Analyze beacon characteristics — interval, jitter, payload size. If suspicious, isolate the endpoint, capture traffic, investigate what's running. Could be malware or could be legitimate heartbeat from monitoring software."

Detailed (60 sec)
"30-second beaconing is highly suspicious — could be C2, could be heartbeat. Need to prove either way.

PHASE 1 — CONFIRM THE PATTERN:
• Query network logs: Connections to same IP, 30-sec intervals
• How long has this been happening?
• Only one host or multiple?
• Consistent interval suggests automation (malware or legitimate)

PHASE 2 — THREAT INTEL:
• Check destination IP/domain against:
  - MDTI (Microsoft Defender TI)
  - VirusTotal
  - Internal blocklists
• Known malicious = high confidence, respond immediately

PHASE 3 — ENDPOINT CORRELATION:
• Which process is making connections?
• Query MDE: DeviceNetworkEvents for that IP
• Get process name, command line, file path
• Check file hash against known malware

PHASE 4 — BEACON ANALYSIS:
• Perfect 30-sec interval = scripted
• Slight jitter (28-32 sec) = sophisticated C2
• Payload size consistent? Data exfil would vary.

PHASE 5 — DECISION TREE:
If process is known (Zoom, Teams, monitoring agent): False positive, document and tune
If process is unknown or suspicious: Isolate endpoint, collect memory dump, block destination at firewall, full incident response

PHASE 6 — HUNTING:
Are other endpoints talking to this IP? Historical search — how long compromised?"
"A vendor-managed ERP system shows signs of compromise, but you don't have direct access. How do you investigate?"
Quick (20 sec)

"Limited access means network-centric investigation. What do we have? Firewall logs for connections, proxy logs for web traffic, identity logs for user access. Look for anomalies: unusual outbound connections, off-hours access, data transfer patterns. Engage vendor immediately — they need to investigate internally while we monitor the perimeter. Document everything for potential legal action."

Detailed (60 sec)
"Vendor-managed means no endpoint visibility, but we still have network and identity telemetry.

AVAILABLE DATA:
• Firewall logs: Connections to/from ERP system
• Proxy logs: Web traffic if applicable
• Identity: Who's authenticating to ERP
• Application logs: If vendor shares or forwards them

INVESTIGATION APPROACH:

PHASE 1 — NETWORK ANALYSIS:
• Unusual outbound destinations?
• Large data transfers?
• Connections to known-bad IPs?
• New ports or protocols?

PHASE 2 — IDENTITY ANALYSIS:
• Unusual login times or locations?
• New accounts accessing system?
• Privilege escalation indicators?
• Service account anomalies?

PHASE 3 — VENDOR ENGAGEMENT (IMMEDIATE):
• Contact vendor security team
• Provide IOCs we've observed
• Request their internal investigation
• Define communication cadence

PHASE 4 — CONTAINMENT OPTIONS:
• Network segmentation: Restrict ERP traffic flows
• Identity: Force password reset for ERP users
• Block specific IOCs at perimeter
• Can't touch the system, but can limit damage

PHASE 5 — DOCUMENTATION:
Critical for potential legal/contract action. Timeline of observations, communications with vendor, business impact assessment.

PHASE 6 — LESSONS LEARNED:
Log integration requirements for vendor contracts, monitoring capabilities for managed systems, segmentation improvements."

Architecture & Design Scenarios

"You're designing SIEM architecture for a new company with 5,000 endpoints, hybrid cloud, and regulatory requirements (SOX, PCI). What's your approach?"
Quick (20 sec)

"Start with compliance requirements — they drive retention and audit needs. XDR-first for endpoint and identity detection. Sentinel for compliance logs, network, and cross-domain correlation. Storage tiering: Analytics for active detection, Basic for compliance search, Archive for long-term. Single workspace with proper RBAC. Design for commitment tier from Day 1."

Detailed (90 sec)
"5,000 endpoints, hybrid, SOX/PCI — this is a structured approach:

PHASE 1 — REQUIREMENTS GATHERING:
• Compliance: SOX needs audit trails, PCI needs cardholder data environment monitoring
• Retention: SOX typically 7 years, PCI 1 year minimum
• Data residency: Where can data live?
• Budget: Drives tiering decisions

PHASE 2 — ARCHITECTURE DECISIONS:
Workspace: Single workspace, multi-region DCRs if needed
Detection split: XDR for endpoint/identity, Sentinel for rest
Tiering:
• Analytics: Security events with active detections
• Basic: Compliance/audit logs (search-only)
• Archive: Long-term retention for SOX

PHASE 3 — LOG SOURCE PRIORITIZATION:
Priority 0 (Week 1-2): Identity (Entra ID) — SOX audit trail, XDR connector — detection foundation
Priority 1 (Week 3-4): PCI zone: Firewalls, card data environment, Domain controllers
Priority 2 (Month 2): Network infrastructure, SaaS applications, Custom applications

PHASE 4 — DETECTION STRATEGY:
• Content Hub for baseline detections
• PCI-specific rules: Card data access, PCI zone violations
• SOX-specific: Privileged access, separation of duties

PHASE 5 — COMPLIANCE MAPPING:
• Each requirement mapped to log source + detection
• Evidence of monitoring for auditors
• Workbooks for compliance reporting

PHASE 6 — COST MODELING:
• Estimate volume per source
• Design for commitment tier threshold
• Project 12-month cost before implementation

DELIVERABLE: Architecture document with diagram, cost model, timeline, compliance mapping, and detection catalog."
"Leadership wants to consolidate from three SIEMs to one. How do you approach the migration?"
Quick (20 sec)

"Don't do big bang. Parallel operation is mandatory. Inventory all three — log sources, detections, integrations, users. Find the overlap and unique value of each. Migrate detection coverage, not rules 1-to-1. Phase by business criticality, not by source SIEM. Maintain all three until target has proven coverage. Gate-based cutover, not calendar."

Detailed (90 sec)
"Three SIEMs to one is high risk. Methodical approach:

PHASE 1 — DISCOVERY (4-6 weeks):
For each SIEM:
• Log sources: What's ingested, volume, owner
• Detections: Rule count, firing frequency, value
• Integrations: Ticketing, SOAR, reporting
• Users: Who uses it, for what
• Cost: Licensing, infrastructure, operations

PHASE 2 — RATIONALIZATION:
• Overlap: Same logs in multiple SIEMs?
• Unique value: What does each SIEM do that others don't?
• Detection redundancy: Same threats detected multiple ways?
• User consolidation: Can workflows merge?

PHASE 3 — TARGET ARCHITECTURE:
• Which platform is target? (Assume Sentinel based on strategy)
• What moves vs what retires vs what stays (maybe Splunk for ops)
• Detection ownership: XDR vs SIEM split

PHASE 4 — MIGRATION WAVES:
Wave 1: Highest-value, lowest-risk sources
Wave 2: Complex sources, custom parsers
Wave 3: Legacy, edge cases
Each wave: Ingest → Detect → Validate → Cutover source

PHASE 5 — PARALLEL OPERATIONS:
• All SIEMs running during migration
• Duplicate critical detections temporarily
• Compare alert quality between platforms
• SOC trained on target before any cutover

PHASE 6 — CUTOVER CRITERIA (per SIEM):
• All log sources migrated
• Detection parity confirmed
• 90 days stable
• SOC sign-off
• Then decommission

TIMELINE: 12-18 months realistic. Don't promise 6 months."

Troubleshooting Scenarios

"MDE alerts aren't appearing in Sentinel. How do you troubleshoot?"
Quick (20 sec)

"Check the pipeline: M365 Defender connector enabled and healthy? Permissions correct — SecurityReader minimum? Check SecurityAlert and SecurityIncident tables for recent data. If connector shows healthy but no data, check filtering in connector settings. If data exists but alerts missing, check if XDR is actually generating alerts — might be suppressed or filtered upstream."

Detailed (60 sec)
"MDE to Sentinel is a pipeline. Troubleshoot each step:

STEP 1 — CONNECTOR STATUS:
• Data connectors > Microsoft 365 Defender
• Status should be 'Connected'
• Check last data received timestamp
• Review any error messages

STEP 2 — PERMISSIONS:
• Service principal or user needs SecurityReader
• Check Azure AD app registration if using service principal
• Verify tenant consent granted

STEP 3 — DATA VERIFICATION:
SecurityAlert | where ProviderName == 'MDATP' | where TimeGenerated > ago(24h) | count
SecurityIncident | where TimeGenerated > ago(24h) | count

STEP 4 — CONNECTOR CONFIGURATION:
• Which data types enabled? Alerts? Incidents? Raw events?
• Filtering applied? Maybe filtering out what you need
• Check the specific products enabled (MDE, MDI, MDO, MDA)

STEP 5 — UPSTREAM VERIFICATION:
• Go to security.microsoft.com
• Are alerts actually generating in XDR?
• If no alerts in XDR, problem is MDE, not connector

STEP 6 — COMMON ISSUES:
• Connector disabled after tenant change
• Permissions revoked
• Filtering too aggressive
• XDR alerts suppressed by automation rules
• Time sync issues between tenants

STEP 7 — RESOLUTION:
Usually reconnecting with proper permissions fixes it. If persistent, open Microsoft support ticket."
"An ASR rule is blocking a legitimate business application. How do you handle it?"
Quick (20 sec)

"First, confirm it's actually legitimate — verify with app owner. Then create targeted exclusion — file path, hash, or certificate — whatever is most specific. Document the business justification and get sign-off. Never disable the entire rule; just exclude the specific legitimate use. Test exclusion before broad deployment."

Detailed (60 sec)
"ASR blocking legitimate app is expected during rollout. This is why we run audit mode first.

STEP 1 — VERIFY LEGITIMACY:
• What application? Who owns it?
• Expected behavior or user workaround attempt?
• Get confirmation from app owner/IT

STEP 2 — UNDERSTAND THE BLOCK:
• Which ASR rule triggered?
• What specific action was blocked?
• Review in M365 Defender > ASR rules report
• Get: File path, process name, hash

STEP 3 — DESIGN EXCLUSION:
Options (most specific to least):
1. Certificate-based: If app is signed, trust publisher
2. File hash: Specific version of file
3. File path: Last resort, be as specific as possible

Avoid: Broad folder exclusions, disabling rule entirely

STEP 4 — DOCUMENTATION:
• Business justification documented
• App owner sign-off
• Risk acceptance if needed
• Expiration/review date

STEP 5 — IMPLEMENTATION:
• Test exclusion in pilot group first
• Deploy via Intune policy
• Verify app works
• Verify rule still blocks unknown threats

STEP 6 — COMMUNICATE:
• Update change record
• Notify SOC of new exclusion
• Add to monthly exclusion review list

PRINCIPLE: Every exclusion is an accepted risk. Document and review regularly."
"You're seeing 30% of endpoints not reporting to MDE. How do you troubleshoot?"
Quick (20 sec)

"30% is significant. Segment the problem: Check device inventory by OS, department, location. Common causes: Onboarding failed, connectivity issues (proxy/firewall), conflicting security software, machines offline. Use Intune compliance reports and MDE device health. For persistent issues, run MDE Client Analyzer on sample devices. Usually it's network connectivity or deployment gaps."

Detailed (60 sec)
"30% gap is a major coverage issue. Systematic approach:

STEP 1 — IDENTIFY THE 30%:
• Export device inventory from Intune and MDE
• Compare: What's in Intune but not in MDE?
• Segment by: OS, department, location, device type

STEP 2 — COMMON CAUSES:

Onboarding Issues:
• Package never deployed (Intune policy not assigned)
• Deployment failed (check Intune deployment status)
• Unsupported OS version

Connectivity Issues:
• Proxy blocking MDE URLs
• Firewall rules blocking outbound
• VPN split tunnel misconfiguration
• Machines on isolated network

Conflicting Software:
• Old AV not removed
• Security software conflict
• EDR coexistence not configured

Offline/Stale:
• Machines powered off
• Employees on leave
• Decommissioned but not removed from inventory

STEP 3 — DIAGNOSTIC TOOLS:
• MDE Client Analyzer on sample devices
• Test connectivity to MDE URLs
• Check Windows event logs for sense service
• Intune device compliance status

STEP 4 — REMEDIATION:
For each root cause:
• Onboarding: Re-deploy package
• Connectivity: Update proxy/firewall rules
• Conflicts: Remove conflicting software
• Stale: Clean up inventory

STEP 5 — MONITORING:
• Create dashboard for onboarding health
• Alert on devices that stop reporting
• Weekly review of coverage percentage

TARGET: 95%+ coverage before considering EDR cutover."
"Your Sentinel workspace is hitting performance issues — queries timing out, slow dashboards. How do you troubleshoot?"
Quick (20 sec)

"Performance issues usually mean query inefficiency or volume problems. Check: Are queries scanning too much data? Missing time filters? Expensive joins? On the volume side: Any source spiking unexpectedly? Basic tier misused for complex queries? Solutions: Optimize queries with summarize and project early, use materialized views for dashboards, consider ADX for heavy analytics."

Detailed (60 sec)
"Sentinel performance issues have predictable causes:

DIAGNOSIS STEP 1 — IDENTIFY THE PROBLEM:
• Which queries are slow? Specific tables?
• Dashboards or ad-hoc hunting?
• Recent changes in volume?

DIAGNOSIS STEP 2 — COMMON CAUSES:

Query Issues:
• Missing time bounds: Always filter TimeGenerated first
• Table scans: | where after | project = full scan
• Expensive joins: Cross-table joins without time bounds
• Too much data: Querying months when you need hours

Volume Issues:
• Log source spiking: Misconfiguration, attack, debug enabled
• Duplicate ingestion: Double collection path
• Verbose logging: Debug logs in production

Architecture Issues:
• Basic tier for complex queries: Basic is search-limited
• Too many concurrent queries: Dashboard with 20 panels
• Massive tables: Single table with billions of rows

SOLUTIONS:

Query Optimization:
• TimeGenerated filter FIRST in every query
• Project early, reduce columns
• Summarize to reduce row count
• Use let statements for reusable filters

Volume Management:
• Identify spike source, fix at ingestion
• DCR filtering before workspace
• Archive old data

Architecture:
• Materialized views for dashboard data
• Summary tables for common aggregations
• ADX for heavy analytics workloads
• Pre-computed workbooks vs live queries

Monitor: Set up Log Analytics workspace insights to catch issues early."
