Program Structure

The transformation program follows a logical sequence. Identity and Infrastructure migrations started first as the cloud foundation. Security tracks followed once the foundation was ready: SIEM and EDR at M1, then DLP at M6 (delayed due to legal constraints and cross-team coordination). Currently in final handover phase.

Strategic Sequencing

Project Timeline Context

Identity & Infrastructure First: AD ‚Üí Entra ID sync and Azure landing zone/network architecture started before security tracks. You can't migrate security tools to cloud until you have the cloud foundation in place. On-prem AD is synced to Entra ID with future plans to decommission on-prem after complete cloud migration.

SIEM & EDR (M1): Once identity and infrastructure foundation was ready, RSA ‚Üí Sentinel and ESET ‚Üí MDE migrations began. MDE rollout started early to establish telemetry before detection implementations.

DLP Later (M6): DLP migration started at Month 6. DLP needs more groundwork ‚Äî legal alignment, cross-team collaboration, and data classification. Current scope: Symantec Endpoint DLP ‚Üí Purview. Symantec Network DLP continues; plan to discontinue after complete Azure transformation.

Current State: All tracks complete. Now in final handover phase with knowledge transfer to internal security team.

Implementation Order & Dependencies

Critical Understanding: Microsoft's security products have dependencies. The order matters:

  1. Sentinel workspace first ‚Äî All connectors need a destination
  2. M365 Defender connector early ‚Äî Gets XDR data flowing to Sentinel (FREE)
  3. MDE before MDI ‚Äî Endpoint context enriches identity alerts
  4. MDI before MDCA ‚Äî User risk scores inform MDCA policies
  5. All XDR products before detection engineering ‚Äî Need data flowing to build detections
  6. DLP parallel track ‚Äî Least dependent on other products

Team Structure & Roles

The program requires clear separation between platform engineering (building the foundation) and detection engineering (building the rules). This structure avoids bottlenecks and enables parallel progress.

Role Scope: Security Engineer (This Documentation)

SIEM Track: Log source identification, stakeholder coordination, ingestion strategy, Sentinel workspace setup, connector configuration, custom log onboarding. Enabled built-in detection rules from Content Hub, configured XDR connectors, aligned endpoint telemetry correlation.

EDR Track: MDE deployment, policy configuration, ASR rollout, exclusions management, coexistence handling, operational tuning. Ensured consistent telemetry flow and detection alignment.

Parallel Operations: Worked both SIEM and EDR tracks in parallel after MDE rollout began.

DLP Track (Supporting): Audit mode monitoring, tuning, and validation after SIEM/EDR stabilized.

Detection Handoff: Detailed custom KQL detection development and ongoing fine-tuning handled by SOC analysts and dedicated detection engineers.

Detection-Driven Ingestion Strategy

A key architectural decision: delay full Sentinel log ingestion until detections and use cases are defined, while rolling out MDE/XDR in parallel. This controls cost, avoids noise, and ensures telemetry is ingested with purpose.

Why This Matters: Sentinel Is Cost-Driven by Ingestion

Sentinel pricing is either pay-as-you-go or commitment tiers. Blindly ingesting everything early:

  • Burns budget on logs you may not need
  • Creates noise without detections to use the data
  • Produces alerts with no SOC-ready rules
  • Frustrates SOC teams with untuned data

Mature teams avoid this. They design detection coverage first, then ingest only what's needed.

Critical Insight: MDE/XDR Is Independent of Sentinel Ingestion

XDR Works Without Sentinel

MDE telemetry goes directly to the Defender backend. XDR detections work immediately ‚Äî no Sentinel required.

Sentinel ingestion is optional for XDR to function. It's used later for correlation and SIEM workflows.

So it's completely valid to:

  • Roll out MDE agents ‚úÖ
  • Configure ASR, AV, EDR ‚úÖ
  • Stabilize telemetry ‚úÖ
  • Let XDR start detecting immediately ‚úÖ

All without full Sentinel log ingestion yet.

The Phased Ingestion Timeline

PhaseTimelineSentinel IngestionXDR/Detection Work
Phase 1: Foundation Months 0‚Äì2 Minimal ‚Äî identity logs, Azure activity only. Cost kept low. MDE onboarding starts. XDR detections active from Day 1. Policies in audit ‚Üí enforce.
Phase 2: Detection Design Months 1‚Äì3 No new ingestion yet. Focus on designing, not ingesting. Build detection catalog. Map threats ‚Üí MITRE. Decide XDR vs Sentinel ownership. Identify which logs are actually needed.
Phase 3: Controlled Ingestion Months 3+ Enable connectors selectively ‚Äî only logs required by detections. Apply DCR filtering (Event IDs, severity). Detection rules deployed. SOC processes refined. Tune volume based on actual usage.
Phase 4: Steady State Months 6+ Move to commitment tier once daily volume is predictable. Optimize tier placement. XDR handling 780 techniques. Sentinel handling 485 rules. Full coverage operational.

Interview Talking Point

"We delayed full Sentinel log ingestion until our detection catalog was defined. MDE/XDR was rolled out in parallel and started detecting from Day 1 ‚Äî it doesn't need Sentinel to function. During months 1‚Äì3, we built the detection catalog, mapped MITRE coverage, and decided what XDR handles vs what Sentinel needs to own. Only then did we enable selective ingestion ‚Äî each log source tied to a specific detection use case. This kept costs controlled, avoided noisy data, and ensured SOC received purposeful telemetry with ready detections."

Cost of Parallel Operations

Running legacy and modern systems in parallel for 12+ months is expensive. Budget estimates:

  • RSA NetWitness: ~$18,000/month √ó 12 = $216,000
  • ESET licenses (4,500 endpoints): ~$12,500/month √ó 12 = $150,000
  • Symantec DLP: ~$10,000/month √ó 12 = $120,000
  • Sentinel (ramping): ~$4,000-8,000/month = ~$72,000
  • MDE licenses (E5): Already included in M365 E5 licensing

Total parallel operation overhead: ~$560K ‚Äî justified by risk mitigation for a 2,000-employee, 4,500-endpoint environment serving Fortune 500 customers. Siemens corporate standards require zero security gaps during transition.
