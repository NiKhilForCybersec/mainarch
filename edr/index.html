EDR Migration: ESET ‚Üí Microsoft Defender for Endpoint

Track 2 Status: Complete ‚Äî Handover in Progress

Microsoft Defender for Endpoint deployed across 9,000 endpoints. This track started at M1 alongside SIEM. ESET removed, MDE is primary EDR. Currently in final handover phase with knowledge transfer to internal security team.

My Role in EDR Track

Platform Engineering: Endpoint onboarding and stabilization, MDE policy configuration, agent rollouts via Intune/SCCM, attack surface reduction (ASR) policy deployment, exclusions and allow-listing, coexistence management, and operational tuning to minimize business impact.

Detection Alignment: Ensured consistent telemetry flow from MDE into Sentinel. Aligned endpoint detection use cases with identity and network signals. Worked both SIEM and EDR tracks in parallel from M1.

Detection Engineering Team: Custom endpoint detection development and ongoing tuning handled by SOC analysts and detection engineers. Leveraged Microsoft's extensive built-in detection library.

Current State

MDE: Fully deployed to all 9,000 endpoints. ASR rules in enforcement mode.

XDR Integration: M365 Defender XDR enabled from Day 1. XDR handles endpoint detection natively; Sentinel receives XDR incidents for correlation with network and Azure sources.

Legacy: ESET removed from endpoints. Protect server decommissioned.

Current Phase: Knowledge transfer and handover to internal security team.

XDR-First Detection Strategy

Key Decision: We don't rebuild endpoint detections in Sentinel that XDR handles natively. MDE has kernel-level visibility, AMSI integration, and behavioral analysis that KQL queries can't match.

Result: 780 RSA rules retired to XDR (endpoint: 478, identity: 198, email: 104). Sentinel focuses on network, cloud, SaaS, and higher-order correlation with 485 rules (255 migrated + 230 new coverage).

XDR Works Independently of Sentinel Ingestion

Critical Architectural Point: MDE telemetry goes directly to the Defender backend ‚Äî XDR detections work immediately, no Sentinel required.

This enabled our phased approach: roll out MDE early (M1+), get XDR detecting from Day 1, while keeping Sentinel log ingestion minimal until the detection catalog was designed. We didn't pay for logs we weren't using yet.

Critical Concept: You Configure Controls, Not Detections

The paradigm shift from SIEM to XDR:

In MDE/XDR, Microsoft owns detection logic (behavior-based + ML, continuously updated). You do NOT write malware rules, ransomware rules, or credential theft rules.

You configure security controls: ASR rules, AV settings, sensor coverage, exclusions. These are preventive controls, not detection rules.

üëâ XDR Detection Model ‚Äî The Paradigm Shift ‚Üí

XDR Owns Detection, You Own Trust Decisions

Microsoft does not force blind enforcement. XDR owns the detection logic ‚Äî continuously updated, threat-intel-backed, behavioral. But the organization owns trust decisions: exclusions, scope, and blast-radius control.

False positives are expected early. Failure to plan for them is the real risk. We used audit mode for 4-8 weeks per ASR priority wave, built known-good baselines before enforcement, and documented every exclusion with business justification.

Not Everything Can Run MDE

The goal is security visibility, not agent deployment percentage. For systems where MDE can't be deployed ‚Äî legacy OS, network devices, OT/IoT, vendor-managed systems ‚Äî we implement compensating controls:

  • Defender for IoT: Passive network monitoring for OT/IoT
  • Syslog ‚Üí Sentinel: Log ingestion from network devices and appliances
  • Network segmentation: Isolate unmanaged devices
  • Defender for Cloud Apps: CASB for SaaS visibility

Why MDE Rollout Started Early

Telemetry-First Strategy

Identity & Infrastructure Started First: Cloud foundation (AD ‚Üí Entra ID, Azure landing zone) was already underway before security tracks began. You can't migrate security tools until the cloud foundation is in place.

SIEM & EDR Started at M1: Once foundation was ready, both tracks began. MDE rollout was prioritized early to establish endpoint telemetry before detection implementations.

DLP Started Later (M6): DLP migration began later due to legal constraints and cross-team coordination requirements.

Result: By the time detection work progressed, sufficient endpoint telemetry was already available for validation and tuning.

Migration Approach: Active-Active

Unlike SIEM where parallel means "both logging," EDR active-active means both engines actively protecting. This requires careful exclusion management but provides zero protection gaps.

Timeline

Gate-Based Cutover (Not Calendar-Based)

MDE became primary EDR only after meeting these criteria:

  • 70-80% endpoint onboarding coverage achieved
  • Policy compliance across onboarded devices
  • ASR rules validated in audit mode
  • ESET exclusions properly configured (no conflicts)
  • Detection parity confirmed vs legacy EDR

We ran ESET and MDE in parallel throughout the deployment. This avoided blind spots, validated detection effectiveness, and ensured zero protection gaps during transition.
