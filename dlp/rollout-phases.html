Rollout Phases

DLP Track

The 12-month progression from Audit to full Block mode. Each phase has specific goals, validation gates, and success criteria before advancing.

Phase 1: Audit Mode (Months 3-6)

AspectDetails
Policy ModeTestWithoutNotifications (silent audit)
User ImpactNone — users don't see anything
GoalBaseline data, identify false positives, tune policies
SymantecStill active and enforcing

Audit Mode Activities

  • Deploy all policies in audit mode across all locations
  • Weekly review of Activity Explorer for false positives
  • Adjust SIT confidence levels based on data
  • Add exclusions for legitimate business processes
  • Document baseline metrics

KQL: Audit Mode Analysis

KQL - DLP Activity
// DLP policy matches in audit mode
InformationProtectionLogs_CL
| where TimeGenerated > ago(7d)
| where PolicyMode_s == "Audit"
| summarize 
    Matches = count(),
    UniqueUsers = dcount(UserId_s),
    UniqueFiles = dcount(ObjectId_s)
    by PolicyName_s, SensitiveInfoType_s
| order by Matches desc

// Potential false positives (high volume, single user pattern)
InformationProtectionLogs_CL
| where TimeGenerated > ago(7d)
| summarize Matches = count() by UserId_s, PolicyName_s
| where Matches > 50  // High-volume users to investigate
| order by Matches desc

Gate: Exit Audit Mode

Proceed to Warn When
  • ☐ 4+ weeks of audit data collected
  • ☐ False positive rate <20% (after tuning)
  • ☐ No critical business processes flagged incorrectly
  • ☐ User training materials ready
  • ☐ Help desk briefed

Phase 2: Warn Mode (Months 7-9)

AspectDetails
Policy ModeTestWithNotifications (warn)
User ImpactPolicy tips shown, users can override
GoalBuild awareness, gather feedback, final tuning
SymantecStill active (backup protection)

User Experience in Warn Mode

Users see policy tips but can proceed:

  • Email: MailTip appears before sending, user can override
  • SharePoint: Warning banner on upload, sharing still works
  • Teams: Message flagged with policy tip
  • Endpoint: Toast notification, action proceeds

Gate: Exit Warn Mode

Proceed to Block When
  • ☐ 8+ weeks in Warn mode
  • ☐ Override rate <15%
  • ☐ User training 100% complete
  • ☐ False positive rate <10%
  • ☐ Executive sign-off obtained

Phase 3: Block with Override (Months 10-11)

AspectDetails
Policy ModeEnable with override allowed
User ImpactBlocked by default, business justification to proceed
GoalNear-full enforcement with escape valve
SymantecDisabled (Purview is primary)

Override Configuration

PowerShell
# Update policy to Block with Override
Set-DlpComplianceRule -Identity "Block Customer Data" `
    -BlockAccess $true `
    -AllowOverride "WithJustification" `
    -OverrideOptions @(
        "FalsePositive",
        "BusinessJustification"
    )

Phase 4: Full Block (Month 12+)

AspectDetails
Policy ModeEnable - hard block
User ImpactAction blocked, no override available
GoalFull data protection enforcement
SymantecDecommissioned
Executive Exception Process

Even in full Block mode, establish an emergency exception process for legitimate business needs. This should require manager + security approval and be time-limited.

Monitoring Dashboard

KQL - DLP Dashboard Queries
// DLP violations by policy and action
InformationProtectionLogs_CL
| where TimeGenerated > ago(30d)
| summarize 
    Blocked = countif(ActionTaken_s == "Block"),
    Warned = countif(ActionTaken_s == "Warn"),
    Overridden = countif(ActionTaken_s == "Override"),
    Audited = countif(ActionTaken_s == "Audit")
    by PolicyName_s
| render barchart

// Override justifications review
InformationProtectionLogs_CL
| where TimeGenerated > ago(7d)
| where ActionTaken_s == "Override"
| project TimeGenerated, UserId_s, PolicyName_s, Justification_s
| order by TimeGenerated desc
