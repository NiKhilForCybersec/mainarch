Enterprise Network Architecture

Understanding the evolution from traditional WAN (MPLS, VPN) to modern architectures (SD-WAN, SASE, ZTNA). This reference covers when to use each technology, how they interconnect, and security implications.

The Big Picture

Traditional: MPLS for private WAN, VPN for remote access, traffic backhauled to data center

Modern: SD-WAN for intelligent routing, SASE for cloud-delivered security, ZTNA for identity-based access

Hybrid Reality: Most enterprises run both — ExpressRoute/MPLS for critical workloads, SD-WAN/SASE for internet and SaaS

Technology Evolution Timeline

1
MPLS Era (2000s-2010s)
Private WAN circuits between sites. All traffic backhauled to data center. Expensive but reliable and secure.
↓
2
VPN + Internet (2010s)
Remote users connect via VPN. Site-to-site VPN as cheaper MPLS alternative. Still requires data center hairpin.
↓
3
SD-WAN (2015+)
Intelligent routing across multiple links (MPLS + Internet). Direct cloud access. Lower cost, better performance.
↓
4
SASE + ZTNA (2020+)
Security delivered from cloud. Identity-based access replacing network-based. "Never trust, always verify."

Traditional WAN Technologies

MPLS (Multiprotocol Label Switching)

AspectDetails
What It IsPrivate WAN service from telcos. Traffic labeled and routed through carrier's private network — never touches public internet.
How It WorksCarrier assigns labels to packets at ingress, routes based on labels (not IP), strips labels at egress. Creates virtual private network across carrier infrastructure.
Security ModelInherently private — your traffic is isolated from other customers. No encryption needed (though often added). Carrier manages routing.
Use CasesBranch-to-datacenter connectivity, latency-sensitive apps (VoIP, video), guaranteed bandwidth requirements, regulatory compliance.
ProsReliable, predictable latency, SLA-backed, carrier-managed, QoS built-in
ConsExpensive ($500-$2000/site/month), slow provisioning (weeks), inflexible, doesn't help with cloud/SaaS

VPN (Virtual Private Network)

VPN TypeUse CaseHow It Works
Remote Access VPN Employees working from home/travel Client software on laptop connects to VPN concentrator at data center. All traffic tunneled through encrypted connection.
Site-to-Site VPN Connecting branch offices over internet Firewall/router at each site establishes IPsec tunnel. Cheaper than MPLS but no SLA guarantees.
SSL VPN Clientless access to web apps Browser-based access through reverse proxy. No client install needed. Limited to web applications.
VPN Limitations (Why ZTNA Emerged)
  • Full network access: Once connected, user has broad network access — lateral movement risk
  • Backhauling: All traffic routes through data center, even for cloud apps — latency
  • Implicit trust: If you're on VPN, you're "trusted" — no continuous verification
  • Scalability: VPN concentrators become bottlenecks with many remote users

Azure ExpressRoute

What is ExpressRoute?

ExpressRoute = Private connection to Azure (and M365).

Instead of going over public internet to reach Azure, you have a dedicated circuit through a connectivity provider. Think of it as "MPLS to Azure."

AspectDetails
How It WorksYou connect to a Microsoft Enterprise Edge (MSEE) router at an ExpressRoute peering location (colocation facility). Your traffic never traverses public internet to reach Azure.
Connectivity Models
  • Colocation: Your equipment in same facility as MSEE
  • Point-to-Point: Dedicated circuit from your DC to peering location
  • Any-to-Any (IPVPN): Connect via your existing MPLS provider
Peering Types
  • Azure Private Peering: Access VNets, VMs, private endpoints
  • Microsoft Peering: Access M365, Dynamics, Azure public services
Bandwidth Options50 Mbps to 100 Gbps depending on provider and location
RedundancyEach circuit has 2 connections (active-active). For full redundancy, deploy 2 circuits in different peering locations.

Peering Locations: Equinix, Megaport, AT&T, Verizon, etc.
Example: Chicago (Equinix CH1) → Azure East US 2

ExpressRoute vs VPN to Azure

AspectExpressRouteSite-to-Site VPN
ConnectionPrivate circuit via providerIPsec over public internet
BandwidthUp to 100 GbpsUp to 10 Gbps aggregate
LatencyPredictable, lowVariable (internet dependent)
Cost$$$ (circuit + Azure egress)$ (just VPN gateway)
Use CaseProduction workloads, large data transferDev/test, backup, small sites
Setup TimeWeeks (provider provisioning)Hours
When to Use ExpressRoute
  • Large data transfers: Migrating TBs to Azure, backup/DR replication
  • Latency-sensitive apps: Real-time apps, trading systems
  • Compliance: Regulatory requirements for private connectivity
  • High bandwidth: Consistent throughput requirements >1 Gbps
  • Hybrid workloads: Tight integration between on-prem and Azure

SD-WAN (Software-Defined WAN)

What is SD-WAN?

SD-WAN = Intelligent routing across multiple WAN links.

Instead of single MPLS circuit, use multiple connections (MPLS + broadband + LTE) and route traffic based on application requirements, link quality, and cost.

CapabilityHow It Helps
Multi-Link Aggregation Combine MPLS + Internet + LTE. Use all links simultaneously. Failover if one fails.
Application-Aware Routing Route M365 over internet (fast), SAP over MPLS (reliable), backup over LTE (resilience).
Direct Cloud Access Branch users access SaaS directly — no backhaul to data center. Lower latency.
Centralized Management Configure all sites from single console. Push policies, monitor health, deploy changes.
WAN Optimization Deduplication, compression, caching. Improve performance over limited links.

SASE (Secure Access Service Edge)

What is SASE?

SASE = SD-WAN + Cloud-Delivered Security (combined).

Gartner term (2019). Instead of backhauling traffic to data center for security inspection, security services are delivered from the cloud — closer to users and applications.

ZTNA (Zero Trust Network Access)

What is ZTNA?

ZTNA = Application-level access based on identity, not network location.

"Never trust, always verify." Users are granted access to specific applications — not the entire network — after verifying identity, device posture, and context.

VPN vs ZTNA

Traditional VPN
✗Connect to network, then access apps
✗Broad network access once connected
✗Implicit trust after authentication
✗Lateral movement possible
✗Traffic backhauled through DC
ZTNA
✓Authenticate, then connect to specific app
✓Access only to authorized applications
✓Continuous verification (every request)
✓Apps invisible to unauthorized users
✓Direct-to-app via cloud broker
ZTNA Access Flow
VPN MODEL:
User → VPN Concentrator → [FULL NETWORK ACCESS] → Find App → Access App
         └── "You're in the building, go wherever you want"


ZTNA MODEL:
User → ZTNA Broker → Identity Check → Device Posture Check → Context Check

                        ┌───────────────────────────────────────────┘
                        │
                        ▼
              ┌─────────────────────┐
              │  Policy Decision:   │
              │  User: john@corp    │
              │  Device: Compliant  │
              │  Location: Home     │
              │  App: Salesforce    │  → ALLOW (direct connection to app only)
              │  Time: Business hrs │
              │  Risk: Low          │
              └─────────────────────┘
              
        └── "You're authorized for this specific app, nothing else visible"

ZTNA Architecture Types

TypeHow It WorksExample Vendors
Agent-Based (Endpoint-Initiated) Lightweight agent on device connects to broker. Broker authenticates and establishes connection to app. App doesn't need to be internet-exposed. Zscaler ZPA, Palo Alto Prisma Access, Cloudflare Access
Service-Initiated (Reverse Proxy) Connector installed in front of app creates outbound tunnel to broker. User accesses via browser — no agent needed. Zscaler ZPA, Entra Private Access, Cloudflare Access

Microsoft Entra Private Access (ZTNA)

Microsoft's ZTNA Solution

Entra Private Access (part of Global Secure Access) provides ZTNA for on-premises and private apps:

  • Deploy connectors in front of private apps (no public exposure needed)
  • Users access apps through Entra — Conditional Access policies apply
  • Integrates with existing Entra ID, MFA, device compliance
  • Replaces VPN for private app access

Decision Framework: When to Use What

ScenarioRecommended TechnologyWhy
Branch office to data center (critical apps) MPLS or ExpressRoute SLA-backed, predictable latency
Branch office to cloud (SaaS) SD-WAN + SASE Direct cloud access, no backhaul
Remote worker to M365 SASE (SWG + CASB) Security inspection, optimized routing
Remote worker to private app ZTNA No VPN, identity-based access
Hybrid cloud connectivity ExpressRoute + VPN backup Private + redundancy
Multi-cloud connectivity SD-WAN with cloud gateways Unified policy across clouds
Small branch / retail SD-WAN (Internet only) Cost-effective, simple
High-security environment ExpressRoute + Private Link No internet exposure

Industry Best Practices

Network Architecture Best Practices

Gartner SASE Recommendations:

  • Converge networking and security — avoid point products
  • Adopt ZTNA to replace VPN for remote access
  • Use cloud-delivered security for distributed workforces
  • Prioritize vendors with global PoP coverage

Microsoft Zero Trust Network Guidance:

  • Segment networks — never flat network architecture
  • Encrypt all traffic, even internal
  • Use identity as primary security perimeter
  • Implement micro-segmentation for critical workloads
