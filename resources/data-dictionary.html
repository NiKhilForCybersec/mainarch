Data Dictionary & Field Mappings

SIEM Track Reference Document

Comprehensive field mappings from RSA NetWitness to Microsoft Sentinel. This document serves as the authoritative reference for log normalization, query translation, and detection rule migration.

1. Data Model Overview

Schema Differences

RSA NetWitness uses a meta-key based schema where events are parsed into standardized meta keys (e.g., ip.src, user.dst).

Microsoft Sentinel uses table-based schemas where each data source has its own table with specific columns (e.g., SigninLogs.IPAddress, DeviceProcessEvents.AccountName).

1.1 Schema Comparison

AspectRSA NetWitnessMicrosoft Sentinel
Data Model Unified meta-key schema Table-per-source schema
Query Language NetWitness Query Language (NWQL) Kusto Query Language (KQL)
Normalization Built-in via parsers ASIM (Advanced Security Information Model)
Time Field time or event.time TimeGenerated
Device Identifier device.type, device.class Table name (e.g., DeviceProcessEvents)

4. Query Translation Examples

4.1 Basic Query Patterns

NWQL → KQL: Failed Logins from External IP
// RSA NetWitness Query Language (NWQL)
device.type = 'winevent_nic' && 
event.cat = 'authentication' && 
result = 'failure' && 
ip.src exists && 
ip.src != 10.0.0.0/8 && 
ip.src != 172.16.0.0/12 && 
ip.src != 192.168.0.0/16

// Microsoft Sentinel KQL
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType != "0"  // Non-zero = failure
| where IPAddress !startswith "10." 
    and IPAddress !startswith "172.16." 
    and IPAddress !startswith "192.168."
| project TimeGenerated, UserPrincipalName, IPAddress, ResultType, ResultDescription, AppDisplayName
NWQL → KQL: PowerShell Execution with Encoded Command
// RSA NetWitness Query Language (NWQL)
device.type = 'winevent_nic' && 
filename.dst = 'powershell.exe' && 
param.dst contains '-enc'

// Microsoft Sentinel KQL
DeviceProcessEvents
| where TimeGenerated > ago(24h)
| where FileName =~ "powershell.exe"
| where ProcessCommandLine has_any ("-enc", "-e ", "-encodedcommand")
| project TimeGenerated, DeviceName, AccountName, ProcessCommandLine, InitiatingProcessFileName
NWQL → KQL: Firewall Denied Traffic to Sensitive Ports
// RSA NetWitness Query Language (NWQL)
device.type = 'paloalto' && 
action = 'deny' && 
ip.dstport = 22,23,3389,445,135,139

// Microsoft Sentinel KQL
CommonSecurityLog
| where TimeGenerated > ago(24h)
| where DeviceVendor == "Palo Alto Networks"
| where DeviceAction == "deny"
| where DestinationPort in (22, 23, 3389, 445, 135, 139)
| project TimeGenerated, SourceIP, DestinationIP, DestinationPort, DeviceAction
NWQL → KQL: User Added to Privileged Group
// RSA NetWitness Query Language (NWQL)
device.type = 'winevent_nic' && 
msg.id = 4728,4732,4756 && 
group contains 'admin'

// Microsoft Sentinel KQL
SecurityEvent
| where TimeGenerated > ago(24h)
| where EventID in (4728, 4732, 4756)
| where TargetUserName has "admin" or TargetUserName has "Domain Admins"
| project TimeGenerated, Computer, SubjectUserName, TargetUserName, Activity

// Alternative using Entra ID AuditLogs
AuditLogs
| where TimeGenerated > ago(24h)
| where OperationName has "Add member to role"
| extend TargetUser = tostring(TargetResources[0].userPrincipalName)
| extend Role = tostring(parse_json(tostring(TargetResources[0].modifiedProperties))[1].newValue)
| where Role has "admin"
| project TimeGenerated, OperationName, TargetUser, Role, InitiatedBy

4.2 Aggregation Query Patterns

NWQL → KQL: Top Talkers by Bytes
// RSA NetWitness Query Language (NWQL)
device.type = 'paloalto' && action = 'allow' 
SELECT ip.src, SUM(bytes.dst) AS total_bytes 
GROUP BY ip.src 
ORDER BY total_bytes DESC 
LIMIT 10

// Microsoft Sentinel KQL
CommonSecurityLog
| where TimeGenerated > ago(24h)
| where DeviceVendor == "Palo Alto Networks"
| where DeviceAction == "allow"
| summarize TotalBytes = sum(ReceivedBytes) by SourceIP
| top 10 by TotalBytes desc
NWQL → KQL: Failed Logins per User
// RSA NetWitness Query Language (NWQL)
device.type = 'azuread' && result = 'failure'
SELECT user.dst, COUNT(*) AS failed_count
GROUP BY user.dst
HAVING failed_count > 10

// Microsoft Sentinel KQL
SigninLogs
| where TimeGenerated > ago(24h)
| where ResultType != "0"
| summarize FailedCount = count() by UserPrincipalName
| where FailedCount > 10
| order by FailedCount desc

5. ASIM Normalization

ASIM = RSA's Unified Schema for Sentinel

Microsoft's Advanced Security Information Model (ASIM) provides schema normalization similar to RSA's meta-key approach. Use ASIM parsers for cross-source queries.

5.1 ASIM Schemas Available

ASIM SchemaParser FunctionSource TablesKey Normalized Fields
Authentication imAuthentication SigninLogs, SecurityEvent, Syslog TargetUsername, SrcIpAddr, EventResult, LogonType
DNS imDns DnsEvents, CommonSecurityLog DnsQuery, DnsResponseName, SrcIpAddr, ResponseCodeName
Network Session imNetworkSession CommonSecurityLog, AzureNetworkAnalytics SrcIpAddr, DstIpAddr, DstPortNumber, NetworkProtocol
Process Event imProcessCreate DeviceProcessEvents, SecurityEvent TargetProcessName, TargetProcessCommandLine, ActorUsername
File Event imFileEvent DeviceFileEvents, SecurityEvent TargetFileName, TargetFilePath, ActorUsername
Registry Event imRegistry DeviceRegistryEvents, SecurityEvent RegistryKey, RegistryValue, ActorUsername
Web Session imWebSession CommonSecurityLog, W3CIISLog Url, HttpRequestMethod, SrcIpAddr, HttpStatusCode

5.2 ASIM Query Example

KQL - ASIM Normalized Authentication Query
// Query all authentication failures across ALL sources using ASIM
imAuthentication
| where TimeGenerated > ago(24h)
| where EventResult == "Failure"
| summarize FailureCount = count() by TargetUsername, SrcIpAddr, EventProduct
| where FailureCount > 5
| order by FailureCount desc

// This single query covers:
// - SigninLogs (Entra ID)
// - SecurityEvent (Windows)
// - Syslog (Linux PAM)
// - Any other source with ASIM parser

6. Field Mapping Validation Checklist

Validation Required

When migrating queries or rules, validate field mappings against actual data. Field names and values may vary based on source configuration.

6.1 Pre-Migration Validation

  • ☐ Identify all RSA meta keys used in existing rules
  • ☐ Map each meta key to Sentinel table/column
  • ☐ Verify data types match (string, int, datetime)
  • ☐ Check for value format differences (IP format, timestamp format)
  • ☐ Identify fields requiring JSON parsing in Sentinel
  • ☐ Note fields that don't have direct equivalents

6.2 Post-Migration Validation

  • ☐ Run parallel queries in both systems for 30 days
  • ☐ Compare result counts and sample records
  • ☐ Validate detection rule alert counts match
  • ☐ Confirm all required fields are populated
  • ☐ Test aggregation queries return similar results
  • ☐ Document any discrepancies and workarounds
