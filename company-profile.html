Company Evolution & Why This Migration

Understanding the Context

This section explains why an on-premises security stack existed and why cloud migration is happening now. Interviewers often ask: "Why legacy tools at a SaaS company?" The answer is that the product is SaaS, but the infrastructure was traditionally on-premises.

The Origin Story (2015-2022)

RevFlow was founded in 2015 as an enterprise analytics startup. Like many B2B SaaS companies of that era, they chose on-premises hosted infrastructure rather than public cloud:

DecisionReasoning (2015 Context)
Co-located Data Center Predictable costs, enterprise customers trusted "dedicated infrastructure," cloud was still maturing for enterprise workloads
VMware Virtualization Industry standard, team had expertise, straightforward capacity planning
On-Prem Active Directory Required for Windows servers, GPO management, established enterprise pattern
Traditional Security Stack RSA NetWitness for SIEM (enterprise customers expected it), ESET for endpoints, Symantec DLP for data protection
Key Point for Interviews

"The product is SaaS, but the infrastructure was on-prem."

RevFlow delivered a SaaS product to customers, but internally hosted it on their own VMware infrastructure in a co-located data center. This is common for enterprise SaaS companies started pre-2018. The security stack (RSA, ESET, Symantec) made perfect sense for this architectureâ€”they were monitoring and protecting on-premises servers, endpoints, and network perimeter.

In 2022, Siemens acquired RevFlow to add revenue intelligence capabilities to their enterprise software portfolio. This triggered several changes:

Why Migration Now (2024-2025)

The Convergence of Factors

Multiple factors aligned in 2024 to make migration the clear choice:

DriverDetails
1. Data Center Lease Expiration Co-location contract ending Q4 2025. Renew for 5 years at increased rates, or migrate to Azure. Business case favored migration.
2. M365 E5 Already Licensed Paying for MDE, MDI, MDO, Sentinel, Purview, MDCA through E5 licensing but not using them. Meanwhile paying separately for RSA, ESET, Symantec.
3. Tool Maintenance Burden 3-person security team spending 40%+ time on tool maintenance: RSA upgrades, ESET server patching, Symantec policy server issues. Cloud-native tools reduce operational overhead.
4. Integration Gaps RSA, ESET, Symantec don't talk to each other. Manual correlation required. Microsoft stack provides native XDR correlation across endpoint, identity, email, cloud apps.
5. Siemens Corporate Alignment Other Siemens subsidiaries already on Microsoft security stack. Enables cross-subsidiary threat intelligence sharing and unified SOC visibility.
6. Remote Work Reality Post-COVID, 60% of workforce remote. VPN + on-prem AD + perimeter security model showing strain. Cloud-native identity and security better suited.

Security Architecture for Healthcare

Building DietIQ requires understanding healthcare-specific security requirements:

RequirementImplementation Approach
HIPAA Compliance PHI encryption at rest/transit, access logging, BAA with cloud providers, minimum necessary access controls
EHR/EMR Integration HL7 FHIR APIs for interoperability, OAuth 2.0/SMART on FHIR for authorization, secure webhook handlers
Multi-Provider Access Role-based access (physician vs dietitian vs patient), consent management, audit trails for all PHI access
Patient Data Sovereignty Data residency controls, patient consent workflows, right to deletion/portability
Wearable Device Security Secure API integrations with Fitbit/Apple Health/Whoop, data minimization, user consent for each data type
Provider Identity NPI verification, credential validation, SSO with hospital identity providers (SAML/OIDC)
